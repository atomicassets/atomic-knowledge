# AtomicMarket contract

Validated behavior of the shared `atomicmarket` contract: attribution, listing semantics, fee application, royalty logging, and the V2 changes that affect integrations.

## Marketplace attribution fields on the shared contract

The `atomicmarket` contract is shared: multiple competing marketplaces list into the same `sales` and `auctions` tables. A `sales` row records only `maker_marketplace` — the taker marketplace is supplied as a parameter to `purchasesale`, which erases the row, so a completed sale's taker can never be attributed from table state. An `auctions` row carries `maker_marketplace`, `taker_marketplace`, and `current_bidder`; an empty `taker_marketplace` is meaningful only when `current_bidder` is non-empty, otherwise it is just the unset default. AtomicAssets `offers` rows carry no marketplace field at all, so standalone offers are unattributable; a sale's backing offer is reachable via the sale row's `offer_id`, while auctions have no backing offer (assets transfer directly to the contract). The empty-string marketplace name is the contract's seeded default marketplace. When reading these tables over `/v1/chain/get_table_rows`, nodeos serializes uint64 values above 2^32 as JSON strings and smaller values as JSON numbers — current `sale_id`/`auction_id`/`offer_id` values arrive as numbers, but asset ids (around 2^40) arrive as strings, so parse id fields defensively rather than assuming one shape.

## Overlapping listings are valid chain state

AtomicMarket instant sales are lazy-accept escrow: `announcesale` records a sale row and moves no assets; the asset stays in the seller's wallet until a buyer executes `purchasesale`, which accepts the underlying AtomicAssets offer, transfers the asset, and erases only that sale's row. Multiple live listings of the same asset are therefore valid chain state — typically stale listings left by previous owners after a transfer or purchase, or listings by one seller covering different asset bundles that share an asset. One constraint applies: `announcesale` rejects a second announcement by the same seller for the exact same set of asset ids (the check hashes the sorted id list, so ordering does not evade it). After the first purchase, remaining overlapping listings become unfulfillable but stay on chain until cancelled; because the seller no longer owns the asset, the contract classifies them as invalid and `cancelsale` can then be called without seller authorization. An indexer showing several listings for one asset is faithfully mirroring the chain, and reconcilers should not delete them as drift.

## Collection fees apply at execution time

AtomicMarket V2 applies the collection fee at execution time: settlement reads the market fee live from the AtomicAssets `collections` row, and the `collection_fee` stored on a sale, auction, or buyoffer row is informational only (emitted by the `lognew*` actions for indexing). A collection author can therefore change the fee between listing and settlement — raises and reductions both take effect immediately on every open listing, bounded by the 15% AtomicAssets maximum. The buyer always pays the listed price; only the seller/collection split moves, so a seller can net less than the fee shown at listing time. Frontends should display the live collection fee at the point of sale, and indexers should attribute fees from settlement-time data (the royalty log actions) rather than the stored listing row. Guard rails: `setmarketfee` and `addbonusfee` reject configurations whose fees plus the 15% maximum exceed the sale price at config time, settlement re-asserts `0 <= fee <= 0.15` on the live value, and the payout function requires a strictly positive seller remainder.

## Royalty logs are trace-only

AtomicMarket V2's royalty log actions (`logroyfound`, `logroytempl`, `logroyattr`, `logroydust`) are inline actions with no `require_recipient` — deliberately, because notifying an arbitrary payout recipient would let a recipient contract abort in its notification handler and block the collection's settlements. A notification-driven indexer therefore never sees them; only a trace-reading pipeline (state history) captures royalty events. The logs carry the final per-recipient payout amounts computed once at settlement, with integer-rounding dust and payee-less shares reported through `logroydust`, so the logged amounts always sum to exactly the collection fee. Indexers should store the logged amounts rather than recomputing the split from the royalty config tables, which can diverge through rounding or config changes between listing and settlement under the execution-time fee model. The contract repo's `docs/api-integration.md` documents the full indexer interface and the VeRT suite is an executable reference for every flow.

## Bundle listings under V2 (legacy retirement)

AtomicMarket V2 removes bundle (multi-asset) listings, and legacy bundle rows created under V1 are treated as inherently invalid. Any account may cancel a legacy bundle sale or auction via `cancelsale`/`cancelauct` — the contract marks a row with more than one asset id as invalid, which waives the seller-auth requirement. A legacy bundle auction that already has a bid can be dissolved (bid refunded, assets returned) only while neither side has claimed; partially claimed bundle auctions must be wrapped up through the claim actions. Bundle buyoffers are different: `cancelbuyo` always requires the buyer's authorization, so nobody else can force-refund them. Calling `acceptbuyo` on a legacy bundle buyoffer does not trade — it cancels the buyoffer and refunds the escrowed price to the buyer, exactly like `declinebuyo`. Purchasing a legacy bundle sale likewise cancels the listing without charging the buyer.

## Defensive guards in the V2 contract

The AtomicMarket V2 contract carries four classes of defensive guard worth knowing when auditing or extending it. First, fee configuration is bounded at config time (`setmarketfee` and `addbonusfee` reject fees that, together with the 15% maximum collection fee, exceed the sale price) with a strict `seller payout > 0` assertion at settlement as the backstop — the only place that sees stacked bonus fees plus the live collection fee together. Second, the 3-iterator `std::is_permutation(first1, last1, first2)` overload never checks the second range's length and reads past the stored vector on a longer caller-supplied list — undefined behavior in wasm linear memory — so `assertsale`, `assertauct`, and `acceptbuyo` use the length-checking 4-iterator overload. Third, the collection fee read live at settlement is re-asserted to `0 <= fee <= 0.15`, because casting a negative double to uint64_t is undefined behavior, not a large number. Fourth, `--table.end()` on a possibly-empty multi_index is guarded by a `begin() != end()` check first. Each guard has regression coverage in the contract repo's VeRT test suite.
