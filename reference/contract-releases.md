# Contract releases and deployment

Validated behavior of the AtomicAssets and AtomicMarket contract build and release pipeline, and of deploying contract code on chain.

## Raw vs patched ABI

The AtomicAssets and AtomicMarket contract repos produce two ABIs per build, and they are not interchangeable. `make build` emits the raw CDT 4.1 ABI, which is what the VeRT/jest test suite asserts against; running the ABI patcher before testing breaks the suite. `make release` additionally runs `scripts/patch-abi.py`, which normalizes exactly two CDT 4.1 spelling changes back to the legacy forms existing integrations expect: `vector<uint8_t>` rendered as `uint8[]` instead of `bytes`, and pair fields named `key`/`value` instead of `first`/`second`. The patched ABI is the artifact that ships in GitHub releases and gets deployed on-chain; the wasm is identical either way, since ABI labels do not affect the binary wire format. CI verifies releases with `patch-abi.py --verify-against RAW PATCHED`, which asserts that only the two sanctioned spelling changes differ and that the patch is idempotent; a known limitation is that the verifier cannot distinguish `vector<int8_t>` (which legitimately stays `bytes`) from `vector<uint8_t>`. A practical corollary: when a release's changes are logic-only, the patched ABI is byte-identical to the previous one and `cleos set contract` skips `setabi`, so only `setcode` appears in explorers.

## setcode resource needs and the deploy permission

Deploying the AtomicAssets V2 contract (a wasm of roughly 242KB) consumes on the order of 62KB of NET and about 5,000us of CPU for the `setcode` transaction. The canonical contract accounts sign deploys with a dedicated `deploy` permission linked only to `eosio::setcode` and `eosio::setabi`, so the deploy key cannot delegate bandwidth or run a PowerUp: if the contract account's own staking has lapsed, the deploy fails with resource errors and cannot self-recover. Check the account's available NET and CPU (`cleos get account`) against those magnitudes before any deploy, and top up from a separately funded account (delegatebw for NET, PowerUp for CPU) when needed; long-idle contract accounts on testnets commonly sit far below these magnitudes after their stakes expire.
